  3Ô∏è‚É£ Why we don‚Äôt directly use `default: {}` or `default: []`:
     - In JavaScript, `{}` and `[]` are reference types.

     - In the future, this can cause unexpected behavior or bugs.
     - The safer approach is:
         default: () => ({})
         default: () => []
     - This ensures a new object/array is created for every document.
     - Yes, `default: {}` or `default: []` may work in many cases,
       but for long-term and future safety, the function approach is considered best practice.

  4Ô∏è‚É£ Important Note (About Validation):
     - We have not applied validation in these examples.
     - The focus here is only on understanding the core schema design concept:
         üëâ type + default
     - You can add validation later if needed.

  5Ô∏è‚É£ Why we didn‚Äôt use `required`:
     - If we set `required: true` on a field,
       the user must provide that field, otherwise the server will throw an error.
     - In that case, the importance of `default` decreases.
     - That‚Äôs why in flexible schema design, `required` is sometimes avoided.

  6Ô∏è‚É£ Where validation is better handled:
     - It‚Äôs best to define the basic structure in the schema.
     - For heavy validation, it‚Äôs better to handle it:
         üëâ In the Controller
         üëâ On the Frontend (Zod, Yup, etc.)
     - This keeps the schema flexible.

  7Ô∏è‚É£ Meaning of `type`:
     - It defines what kind of data the field will accept
       (String, Number, Boolean, Object, Array, etc.)

  8Ô∏è‚É£ Meaning of `default`:
     - If the user does not provide data for a field,
       the server will store that field with an empty/initial value.
     - This keeps the response clean and predictable.
     - Example:
         String ‚Üí ""
         Number ‚Üí 0
         Boolean ‚Üí false
         Object ‚Üí {}
         Array ‚Üí []
